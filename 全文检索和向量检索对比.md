body 走的是传统倒排索引，knn 走的是 HNSW 向量索引


| 维度 | BM25 全文检索 | kNN 语义检索 |
|------|--------------|-------------|
| **底层算法** | 词频（TF）× 逆文档频率（IDF）+ 文档长度归一化 | 向量余弦相似度 + HNSW 近似最近邻搜索 |
| **核心能力** | 找"包含这几个关键词"的 chunk | 找"意思相近"的 chunk |
| **匹配方式** | 字面匹配（"收入增长" ≠ "薪资上调"） | 语义匹配（"收入增长" ≈ "薪资上调"） |
| **索引结构** | 倒排索引（词 → 文档列表） | 向量索引（HNSW 图结构） |
| **是否需要模型** | 否，纯统计算法 | 是，需要 embedding 模型编码 |
| **检索速度** | 极快（毫秒级） | 较快（HNSW 近似检索，十毫秒级） |
| **可解释性** | 高（分数来自各词贡献，可回溯） | 低（黑盒向量距离，不可解释） |
| **冷启动** | 开箱即用，不需要训练 | 依赖预训练 embedding 模型 |
| **长文档** | 有长度归一化，自动打折 | 长文本编码可能稀释语义信息 |
| **多义词/同义词** | 无法处理（"苹果"公司 vs 水果） | 能根据上下文区分语义 |
| **典型失效场景** | 同义不同词（"GDP增长"搜不到"经济增速"） | 同词不同义（"表格"误召回 Excel 相关而非 HTML 相关） |
| **ES 调用方式** | `body` 参数中的 `match` 查询 | `knn=` 参数传入向量 |
| **参数配置** | `size=50`（返回前 50 条） | `k=50`（返回前 50 条）、`num_candidates=100`（粗筛候选数） |
| **在项目中的角色** | 第一路召回，保证关键词精确命中 | 第二路召回，补全语义覆盖盲区 |
两路互补——BM25 抓关键词扎堆的，kNN 抓意思接近的。后面 RRF 把两路结果融合，互相补对方的盲区。
